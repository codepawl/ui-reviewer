Install UXRay in the agent loop.

MCP first, API ready. UXRay renders your frontend with Chrome/Playwright, returns screenshots, layout metrics, structured UX issues, and a concrete repair plan, then Codex or Claude Code repairs the UI and asks UXRay to score the diff.

Quick facts

Best first run

npm run demo:pipeline runs baseline eval, asks Codex to repair, reruns eval, and writes the report.

Real proof

The public gallery uses real screenshots from the same local eval pipeline: average score 39 → 100.

MCP tools

review_ui_url includes layout metrics, anti-slop checks, repair steps, and review_ui_diff gates.

Payment

/v1/billing/checkout creates a Creem checkout session for Pro, Team, or review credits.

Fair demo setup

Same fixture family, reviewer, viewport contract, and repair-loop prompt style. Baseline 127.0.0.1:5182; after 127.0.0.1:5183.

Anti-slop gates

Contrast, overflow, sticky overlays, tap targets, hierarchy, copy specificity, decorative gradients, nested cards, one-sided accent cards, and repeated AI-marketing cadence are checked before a repair is treated as shippable.

1. Install locally

git clone https://github.com/codepawl/uxray uxray
cd uxray
npm install
npm run typecheck
npm run build

If the repo is still private while testing, use the local project folder directly when registering MCP clients.

2. Add to Codex

codex mcp add uxray -- npm --silent --prefix /absolute/path/to/uxray run mcp
codex mcp list
codex mcp get uxray

Prompt Codex with a narrow repair contract, or install the skill below so Codex knows when to auto-trigger this loop:

Use the uxray MCP server.
Call health_check.
Call review_ui_url on http://127.0.0.1:5173 with return_images=true, viewport desktop+mobile, strictness high.
Inspect the screenshots, layout metrics, structured issues, and repair_plan.
Fix only the UI files needed.
Call review_ui_url again and compare before/after with review_ui_diff.

3. Add to Claude Code

claude mcp add uxray -- npm --silent --prefix /absolute/path/to/uxray run mcp
claude mcp list
Use the uxray MCP server to review my local frontend URL.
Use screenshot evidence, high-severity issues, layout metrics, and repair_plan as the edit contract.
After editing, rerun the review and summarize the score delta.

4. Install the UXRay agent skill

The skill tells coding agents when to call UXRay, how to avoid vague beautification, and how to complete the review → repair → diff loop without silly mistakes.

mkdir -p ~/.config/uxray/skills
curl -fsSL https://useuxray.com/plugins/uxray-agent-skill.md \
  -o ~/.config/uxray/skills/uxray-agent-skill.md

Skill rule of thumb: auto-trigger UXRay before shipping any generated landing page, dashboard, onboarding flow, settings page, checkout page, docs page, or pricing page.

5. Use the local API

npm run api
curl http://127.0.0.1:4317/health
curl -X POST http://127.0.0.1:4317/v1/reviews/url \
  -H 'content-type: application/json' \
  -d '{"url":"http://127.0.0.1:5173","use_vision":false,"return_images":false}'

Compare two saved reports:

curl -X POST http://127.0.0.1:4317/v1/reviews/diff \
  -H 'content-type: application/json' \
  -d @reports/demo/diff-request.json

7. Updates

UXRay checks the hosted update endpoint and gives local users a command-based upgrade path. Interactive shells can choose auto-upgrade or cancel.

npm run check:update
# choose [u]pgrade or [c]ancel

npm run check:update -- --auto
# non-interactive auto-upgrade

npm run upgrade
# git pull --ff-only, npm install, npm run build

9. Preference learning

UXRay can collect pairwise preference labels instead of pretending there is one universal taste. Start with a simplicity, balanced, or complexity profile, then use public polls, expert notes, and model judges as separate evidence streams.

npm run annotation:pairwise -- \
  --axis spacing_rhythm \
  --profile simplicity \
  --a-label "tight code-block gap" \
  --b-label "28px breathing room"

npm run annotation:ingest:x -- \
  --task data/annotations/tasks/<task_id>.json \
  --post-id 1234567890 \
  --a 42 --b 71 --tie 5
Public annotation

X polls should ask pairwise questions like “which version is easier to scan?” and store only aggregate counts, not voter identities.

Dynamic schema

The dataset keeps screenshots, route type, metrics, taste profile, labels, source confidence, and derived targets for filters or model training.

User taste setup

taste_profile lets reviews bias toward simplicity, balanced defaults, or richer complexity when the information architecture can support it.

10. Pricing direction

Local MCP stays free. Hosted rendering is priced by review calls and bundled for teams. Payment goes through Creem checkout sessions. The paid API runs through Cloudflare with a Fly.io Playwright worker for browser rendering.

POST /v1/billing/checkout
# plan=pro     -> Creem UXRay Pro checkout
# plan=team    -> Creem UXRay Team checkout
# plan=credits -> Creem one-time review credits checkout
